Arena War

An iterative coding benchmark for LLM spatial-strategy algorithms.

Arena War compares how recent language models improve a territory algorithm over repeated rounds of adversarial feedback. Each model writes code, plays against shared opponents, reads the reward signal, and tries again. We report the learning curve, the failure taxonomy, and a representative board so you can see what actually happened.

How To Read This Run
What counts as a good result?

A strong model either finishes with a better score than its peers, improves faster across iterations, or both under the same protocol.

Why trust the comparison?

All models in this run are judged under the same shared setup: same baseline opponents, same iteration budget, and the same scoring loop.

What is the main signal?

The learning curve is primary. Final score matters, but how a model improves after feedback is the benchmark's core capability signal.

What is secondary?

The arena replay is a follow-on inspection tool. Use it after you understand the benchmark result, not instead of it.

What This Benchmark Measures

Arena War measures iterative algorithm improvement under adversarial pressure: a model must read the task, write code, learn from reward signals, and produce stronger follow-up algorithms over repeated rounds. It does not claim to measure general coding ability from one chart. Cross-model comparison is meaningful here because the protocol is shared across the run. For deeper methodology, see benchmark-methodology.md.

Figure 1 — Learning Curves

Figure 1: Mean territory percentage by eval iteration, one line per model. Shaded bands show ±1 standard deviation across the N games per iteration. Read this relatively — the main signals are (a) final height at the latest iteration, (b) slope of improvement across iterations, and (c) the width of the band, which tells you how much of the difference between models is noise.

Figure 3 — Failure Taxonomy

How often each model's iterations hit an annotated failure mode: extraction errors, runtime crashes, out-of-bounds exploits, timeouts, regressions, and plateau stalls. Use this to distinguish models that scored well from models that scored well and stayed stable.

Figure 3: Counts are per-iteration occurrences across this run. An iteration may carry multiple flags (for instance, a regression that also stalled). A tall bar is not necessarily worse — it depends which flag it is. A stack of syntax errors means the model failed to emit runnable code; a stack of regressions means the model kept writing runnable code but lost ground to its own earlier attempts.
